### A.3 技术栈对比: MVP vs 成熟

| 方面 | MVP架构 | 成熟架构 |
|------|---------|---------|
| **API Gateway** | FastAPI内置 | Kong/Traefik |
| **服务间通信** | 模块间函数调用 | gRPC |
| **消息队列** | Redis Streams | RabbitMQ/Kafka |
| **编排平台** | Docker Compose | Kubernetes |
| **服务数量** | 1个单体应用 | 6+个微服务 |
| **开发难度** | 低 | 高 |
| **维护成本** | 低 | 高 |
| **适用场景** | 策略 < 50, 个人项目 | 策略 > 50, 团队协作 |

### A.4 迁移路径

1. **阶段1 (MVP → 成长)**:
   - 拆分monitoring-module独立部署
   - 引入Redis Streams替代Redis Lists

2. **阶段2 (成长 → 成熟)**:
   - 拆分strategy-module和trading-module
   - 引入API Gateway (Kong/Traefik)
   - 模块间通信从函数调用迁移到gRPC

3. **阶段3 (成熟)**:
   - 引入RabbitMQ/Kafka
   - 迁移到Kubernetes
   - 实现多租户支持

### A.5 注意事项

⚠️ **MVP阶段不应考虑以下技术:**
- ❌ API Gateway (增加复杂度,MVP阶段不需要)
- ❌ gRPC (函数调用已足够,Protobuf增加学习成本)
- ❌ RabbitMQ/Kafka (Redis Streams满足MVP需求)
- ❌ Kubernetes (Docker Compose足够,学习曲线陡峭)
- ❌ Service Mesh (过早优化)

✅ **何时引入这些技术:**
- ✅ 需要统一认证、限流、路由时 → API Gateway
- ✅ 服务间调用QPS > 1000时 → gRPC
- ✅ 异步事件量 > 10000/分钟时 → RabbitMQ/Kafka
- ✅ 策略容器数量 > 100时 → Kubernetes
- ✅ 需要服务治理时 → Service Mesh

## 附录B: 微服务详细设计 (未来扩展参考)

⚠️ **重要提示**: 本附录内容是系统演进到微服务架构后的详细设计,**MVP阶段无需实现**。

### B.1 微服务边界划分 (成熟阶段)

基于领域驱动设计 (DDD),系统划分为以下微服务:

| 服务名 | 职责 | 优先级 |
|--------|------|--------|
| **market-service** | 行情数据管理、热门币种、自选币种 | P0 |
| **strategy-service** | 策略库管理、策略校验、策略版本控制 | P0 |
| **trading-service** | 交易所账户管理、订单管理、持仓管理 | P0 |
| **runtime-service** | 策略生命周期管理、运行模式控制、进程隔离 | P0 |
| **monitoring-service** | 运行状态监控、日志聚合、告警通知 | P1 |
| **auth-service** (可选) | 用户认证、权限管理 | P1 |

### B.2 微服务间通信 (成熟阶段)

#### B.2.1 同步通信 (gRPC)

| 场景 | 协议 | 说明 |
|------|------|------|
| Service A → Service B | gRPC | 高性能,强类型 |
| 前端 → API Gateway | REST API | 简单直接,易于调试 |

#### B.2.2 异步通信 (事件驱动)

| 事件名 | 生产者 | 消费者 | 用途 |
|--------|--------|--------|------|
| MarketDataUpdated | market-service | runtime-service | 实时行情推送 |
| OrderStatusChanged | trading-service | runtime-service, monitoring-service | 订单状态更新 |
| StrategyStatusChanged | runtime-service | monitoring-service | 策略运行状态变化 |
| StrategyLogGenerated | runtime-service | monitoring-service | 策略日志收集 |
| AlertTriggered | runtime-service | monitoring-service | 告警通知 |

### B.3 微服务详细设计 (核心服务)

#### B.3.1 market-service (微服务版本)

**职责:**
- 管理热门加密货币列表
- 管理用户自选币种
- 实时行情数据获取和分发
- 历史行情数据存储

**核心gRPC API:**
```protobuf
// market.proto

service MarketService {
  // 获取热门币种列表
  rpc GetTrendingCoins(GetTrendingRequest) returns (GetTrendingResponse);

  // 获取自选币种列表
  rpc GetWatchlist(GetWatchlistRequest) returns (GetWatchlistResponse);

  // 添加自选币种
  rpc AddWatchlist(AddWatchlistRequest) returns (AddWatchlistResponse);

  // 获取单个币种行情
  rpc GetTicker(GetTickerRequest) returns (GetTickerResponse);

  // 获取K线数据
  rpc GetKlines(GetKlinesRequest) returns (GetKlinesResponse);
}
```

#### B.3.2 strategy-service (微服务版本)

**职责:**
- 策略文件上传和存储
- 策略代码校验
- 策略版本管理
- 策略元数据管理

**核心gRPC API:**
```protobuf
// strategy.proto

service StrategyService {
  // 上传策略文件
  rpc UploadStrategy(UploadStrategyRequest) returns (UploadStrategyResponse);

  // 获取策略列表
  rpc ListStrategies(ListStrategiesRequest) returns (ListStrategiesResponse);

  // 获取策略详情
  rpc GetStrategy(GetStrategyRequest) returns (GetStrategyResponse);

  // 删除策略
  rpc DeleteStrategy(DeleteStrategyRequest) returns (DeleteStrategyResponse);
}
```

#### B.3.3 trading-service (微服务版本)

**职责:**
- 交易所账户管理
- 交易所API集成
- 订单生命周期管理
- 持仓管理
- 账户余额查询

**核心gRPC API:**
```protobuf
// trading.proto

service TradingService {
  // 下单
  rpc PlaceOrder(PlaceOrderRequest) returns (PlaceOrderResponse);

  // 撤单
  rpc CancelOrder(CancelOrderRequest) returns (CancelOrderResponse);

  // 查询余额
  rpc QueryBalance(QueryBalanceRequest) returns (QueryBalanceResponse);

  // 查询持仓
  rpc QueryPositions(QueryPositionsRequest) returns (QueryPositionsResponse);
}
```
