## 2. 核心用户画像

### 2.1 基本信息

| 属性 | 描述 |
|------|------|
| **用户类型** | 个人投资者 / 量化交易研究者 |
| **年龄范围** | 25-40岁 |
| **职业背景** | 金融从业者、程序员、或对量化交易有兴趣的技术人员 |
| **技术水平** | 具备基础编程能力，能够阅读和编写简单的策略代码 |
| **量化交易经验** | 有一定经验（1-3年），了解基本的量化交易概念和技术指标 |

---

### 2.2 用户特征

#### 技术能力
- 熟悉至少一种编程语言（如Python、JavaScript）
- 能够理解API接口文档并进行对接
- 具备基本的调试和问题排查能力

#### 量化交易能力
- 了解K线、均线、MACD、RSI等常见技术指标
- 了解回测、模拟盘、实盘的概念和区别
- 有过使用第三方量化平台（如TradingView、聚宽等）的经验

#### 资源情况
- 个人项目，无大规模资源投入
- 可以接受一定的学习成本
- 部署环境：本地电脑或小型云服务器

---

### 2.3 用户目标

| 目标类型 | 描述 |
|----------|------|
| **核心目标** | 研究和验证自己的交易策略 |
| **次要目标** | 实现策略的自动化执行 |
| **长期目标** | 建立自己的交易策略库，实现稳定的投资收益 |

---

### 2.4 用户痛点

| 痛点 | 描述 |
|------|------|
| **市场痛点** | 市面上缺少好用、免费、面向多种市场的量化交易系统 |
| **成本痛点** | 商业量化平台收费昂贵，免费平台功能受限 |
| **定制痛点** | 现有平台难以满足个性化的策略开发需求 |
| **数据痛点** | 部分市场（如港股、美股）的数据获取成本高 |

---

### 2.5 用户使用场景

#### 场景1：策略研究
用户想验证一个基于双均线的趋势跟踪策略，需要：
- 查看历史行情数据
- 编写策略代码
- 进行回测验证
- 根据回测结果调整参数

#### 场景2：多策略并行测试
用户开发了3个不同的策略，需要：
- 同时运行这3个策略
- 观察每个策略的实时表现
- 根据表现决定是否继续运行

#### 场景3：实盘交易
用户对某个策略有信心，需要：
- 配置交易所API
- 将策略切换到实盘运行模式
- 实时监控策略表现
- 出现异常时能及时停止

---

### 2.6 用户约束

| 约束类型 | 描述 |
|----------|------|
| **时间约束** | 个人项目，开发时间有限，希望尽快上线MVP |
| **资源约束** | 无大规模团队支持，依赖个人或小团队开发 |
| **成本约束** | 不愿投入大量资金购买商业软件或高级数据服务 |
| **技术约束** | 不擅长复杂的系统运维，希望部署和使用简单 |

---

## 3. 用户非功能需求

| 维度 | 需求 |
|------|------|
| **易用性** | Web界面操作简单，策略模板易于理解和使用 |
| **可靠性** | 策略运行稳定，多进程隔离避免相互影响 |
| **可维护性** | 策略代码清晰，便于后续修改和优化 |
| **可扩展性** | 支持后续扩展港股、美股市场 |
| **安全性** | API Key等敏感信息安全存储 |

---

## 4. 用户价值主张

| 价值点 | 描述 |
|--------|------|
| **免费开源** | 解决商业平台收费昂贵的痛点 |
| **多市场支持** | 未来支持加密货币、港股、美股多种市场 |
| **高度定制** | 完全自定义策略，不受平台限制 |
| **灵活部署** | 支持本地和云服务器部署 |

## 5. 附录

### 5.1 画像假设说明

本用户画像基于以下假设：
- 用户是产品的唯一目标用户（个人投资者）
- 用户具备一定的技术背景，能够接受一定的学习成本
- 用户的核心诉求是策略研究，而非高频交易或复杂的多用户场景

如果在实际开发过程中发现用户特征与画像不符，需要及时更新。
