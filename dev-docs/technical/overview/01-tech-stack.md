# 技术选型

## 1. 选型原则

### 1.1 项目特征分析

| 特征 | 描述 | 技术影响 |
|------|------|----------|
| 单用户系统 | 仅开发者本人使用 | 简化架构，无需多租户 |
| 策略语言 | 用户使用 Python 编写策略 | 后端必须支持 Python 生态 |
| 用户画像 | 熟悉 Python 的开发者 | 降低维护门槛 |
| 并发需求 | 10+ 策略同时运行 | 多进程架构 |
| 实时性 | 行情推送、订单状态同步 | WebSocket 支持 |
| 数据量 | 大量历史 K 线数据 | 时序数据库优化 |
| 部署方式 | 本地或云服务器 | 容器化部署 |

### 1.2 选型优先级

1. **开发效率** > 极致性能（个人项目，时间有限）
2. **生态成熟度** > 技术新颖性（稳定可靠优先）
3. **运维简单** > 功能丰富（单人维护）
4. **统一技术栈** > 最优单点方案（减少心智负担）

## 2. 后端技术栈

### 2.1 编程语言：Python 3.12

**选择理由**：

| 理由 | 说明 |
|------|------|
| 策略兼容 | 用户策略本身就是 Python 代码 |
| 量化生态 | pandas、numpy、ta-lib 等库成熟 |
| 交易所支持 | ccxt 库统一封装 100+ 交易所 |
| 用户熟悉 | 目标用户精通 Python |
| 开发效率 | 动态类型 + 丰富库 = 快速开发 |

**版本选择**：Python 3.12

- 性能提升 ~5%（PEP 659 解释器优化）
- 更好的错误提示信息
- 类型系统增强（TypedDict、泛型改进）

**备选方案对比**：

| 方案 | 优势 | 不选理由 |
|------|------|----------|
| Go | 高性能、并发模型好 | 量化生态弱，策略需跨语言调用 |
| Rust | 极致性能、内存安全 | 学习曲线陡，开发效率低 |
| Java | 企业级稳定 | 量化生态不如 Python |

### 2.2 Web 框架：FastAPI

**选择理由**：

| 特性 | 说明 |
|------|------|
| 高性能 | 基于 Starlette，异步原生支持 |
| 自动文档 | OpenAPI/Swagger 自动生成 |
| 类型安全 | Pydantic 数据验证 |
| WebSocket | 原生支持，适合实时行情推送 |
| 现代化 | async/await 语法，代码简洁 |

**备选方案对比**：

| 方案 | 优势 | 不选理由 |
|------|------|----------|
| Django | 全栈、Admin 后台 | 太重，ORM/Admin 用不上 |
| Flask | 轻量、灵活 | 无异步原生支持，无自动文档 |
| Tornado | 异步、WebSocket | 社区不如 FastAPI 活跃 |

### 2.3 任务调度：APScheduler

**用途**：

- 定时同步历史数据
- 定时检查策略健康状态
- 定时清理过期日志

**选择理由**：

- 轻量级，无需额外服务
- 支持多种触发器（cron、interval、date）
- 可持久化任务到数据库

### 2.4 进程管理：multiprocessing

**用途**：每个策略作为独立进程运行

**选择理由**：

- Python 标准库，无额外依赖
- 进程隔离，单个策略崩溃不影响其他
- 配合 Redis 实现进程间通信

**备选方案对比**：

| 方案 | 优势 | 不选理由 |
|------|------|----------|
| Celery | 分布式任务队列 | 策略是长时间运行的进程，非短任务 |
| Dramatiq | 比 Celery 简单 | 同上 |
| concurrent.futures | 简单 | 线程池受 GIL 限制 |

## 3. 数据层

### 3.1 主数据库：PostgreSQL 16

**存储内容**：

- 策略配置
- 订单记录
- 交易记录
- 用户配置
- 系统日志

**选择理由**：

| 特性 | 说明 |
|------|------|
| 可靠性 | ACID 事务保证 |
| JSON 支持 | JSONB 类型存储策略参数 |
| 扩展性 | TimescaleDB 扩展支持时序数据 |
| 性能 | 复杂查询优化能力强 |

**备选方案对比**：

| 方案 | 优势 | 不选理由 |
|------|------|----------|
| MySQL | 使用广泛 | JSON 支持不如 PostgreSQL |
| SQLite | 零配置 | 并发写入性能差 |
| MongoDB | 文档模型灵活 | K 线是结构化数据，关系库更合适 |

### 3.2 时序扩展：TimescaleDB

**存储内容**：

- 历史 K 线数据
- 实时行情快照
- 策略收益曲线

**选择理由**：

| 特性 | 说明 |
|------|------|
| PostgreSQL 兼容 | 无需学习新语法 |
| 自动分区 | 按时间自动分片 |
| 压缩 | 90%+ 压缩率，节省存储 |
| 聚合优化 | 时间窗口查询性能优异 |

**数据量估算**：

```
BTC/USDT 1分钟 K 线 5 年:
= 5 × 365 × 24 × 60 = 2,628,000 条

单条 OHLCV 约 50 字节:
= 2,628,000 × 50 ≈ 125 MB（压缩后 ~15 MB）

支持 100 个交易对:
= 100 × 15 MB = 1.5 GB
```

**备选方案对比**：

| 方案 | 优势 | 不选理由 |
|------|------|----------|
| InfluxDB | 专业时序库 | 多一个服务运维，查询语法不同 |
| QuestDB | 超高性能 | 生态不如 TimescaleDB 成熟 |
| ClickHouse | OLAP 性能强 | 单机场景过于重量级 |

### 3.3 缓存/消息：Redis 7

**用途**：

| 用途 | 说明 |
|------|------|
| 行情缓存 | 最新价格、深度数据 |
| 进程通信 | 策略进程与主进程通信 |
| 会话存储 | WebSocket 会话管理 |
| 限流 | API 请求频率限制 |

**选择理由**：

- 单一服务满足多种需求
- 毫秒级响应
- 丰富的数据结构（Hash、List、Pub/Sub）

### 3.4 ORM：SQLAlchemy 2.0

**选择理由**：

| 特性 | 说明 |
|------|------|
| 类型提示 | 2.0 版本原生支持 |
| 异步支持 | 配合 asyncpg 驱动 |
| 成熟稳定 | Python ORM 事实标准 |
| 灵活性 | 可降级到原生 SQL |

## 5. 策略引擎

### 5.1 沙箱隔离

**方案**：子进程 + RestrictedPython

| 层级 | 机制 | 说明 |
|------|------|------|
| 代码层 | RestrictedPython | 禁止危险模块导入 |
| 进程层 | 独立子进程 | 资源隔离 |
| 资源层 | resource 模块 | CPU/内存限制 |

**禁止的操作**：

```python
# 禁止列表
- os.system, subprocess
- open (文件操作)
- __import__ (动态导入)
- eval, exec
- socket (网络访问)
```

### 5.2 技术指标：pandas-ta

**选择理由**：

| 特性 | 说明 |
|------|------|
| 纯 Python | 无需编译 C 扩展 |
| 指标丰富 | 130+ 技术指标 |
| pandas 集成 | DataFrame 原生支持 |

**备选方案对比**：

| 方案 | 优势 | 不选理由 |
|------|------|----------|
| TA-Lib | 性能最好 | 需要编译安装，跨平台麻烦 |
| tulipy | 轻量 | 指标数量少 |

### 5.3 数据处理

| 库 | 用途 |
|------|------|
| pandas | 数据清洗、时间序列处理 |
| numpy | 数值计算 |

## 6. 前端技术栈

### 6.1 框架：Vue 3 + TypeScript

**选择理由**：

| 特性 | 说明 |
|------|------|
| 学习曲线 | 比 React 平缓 |
| 单人维护 | 模板语法直观易读 |
| 组合式 API | 逻辑复用更清晰 |
| 生态完善 | Vite、Pinia、VueRouter 官方维护 |

**备选方案对比**：

| 方案 | 优势 | 不选理由 |
|------|------|----------|
| React | 生态最大 | 对单人项目复杂度偏高 |
| Svelte | 性能好、代码少 | 生态不如 Vue 成熟 |

### 6.2 UI 库：Naive UI

**选择理由**：

| 特性 | 说明 |
|------|------|
| Vue 3 原生 | 非 Vue 2 迁移 |
| TypeScript | 完整类型定义 |
| 暗色主题 | 量化系统常用暗色 |
| 轻量 | 按需加载 |

### 6.3 图表：TradingView Lightweight Charts

**选择理由**：

| 特性 | 说明 |
|------|------|
| 专业 K 线 | 金融级图表 |
| 开源免费 | Apache 2.0 协议 |
| 高性能 | Canvas 渲染 |
| 轻量 | ~40KB gzipped |

### 6.4 状态管理：Pinia

- Vue 3 官方推荐
- 比 Vuex 更简洁
- 完整 TypeScript 支持

## 7. 部署方案

### 7.1 容器化：Docker Compose

**选择理由**：

| 特性 | 说明 |
|------|------|
| 单机部署 | 无需 K8s 复杂度 |
| 一键启动 | `docker compose up -d` |
| 环境一致 | 开发/生产环境统一 |

### 7.2 反向代理：Caddy

**选择理由**：

| 特性 | 说明 |
|------|------|
| 自动 HTTPS | Let's Encrypt 自动申请 |
| 配置简单 | 比 Nginx 配置简洁 |
| 性能足够 | 单用户场景绰绰有余 |

**备选方案对比**：

| 方案 | 优势 | 不选理由 |
|------|------|----------|
| Nginx | 性能最好 | 配置复杂，HTTPS 需手动 |
| Traefik | 动态配置 | 单机场景过于复杂 |
